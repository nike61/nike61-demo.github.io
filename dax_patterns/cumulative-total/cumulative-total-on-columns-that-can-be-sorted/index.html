Накопительный итог по сортируемым колонкам

Паттерн показывает, как накапливается показатель по бизнес-порядку категорий: например, продажи Silver, затем Silver плюс Gold, затем результат после добавления Platinum.

бизнес-порядок классов Customer Class Number Sales Amount RT Class сортировка подписей

Как читается итог

Серые столбцы показывают продажи отдельного класса. Синяя линия показывает сумму этого класса и всех классов с меньшим номером.

Sales Amount Sales Amount RT Class

Какую задачу решает

Обычная сумма показывает вклад каждой категории отдельно. Накопительная мера отвечает на вопрос, сколько уже набрано к текущему уровню классификации, если двигаться по заданному бизнес-порядку.

Классы клиентов

Можно увидеть продажи Silver, затем Silver плюс Gold, затем накопленный результат после Platinum и итог после Titanium.

Упорядоченные группы

Та же логика подходит для стадий сделки, уровней риска, приоритета, грейда или диапазона выручки, если у группы есть числовой порядок.

Без ручного сложения

Пользователь не складывает строки таблицы сам и не зависит от алфавитной сортировки подписей вроде Gold, Platinum, Silver, Titanium.

Когда применять

Паттерн нужен только там, где у категорий есть понятное "раньше" и "позже". Текстовая подпись может быть любой, но рядом с ней должен быть стабильный ключ сортировки.

Хороший сценарий

Класс клиента хранится в таблице Customer: пользователь видит Customer[Customer Class], а порядок задает Customer[Customer Class Number]. Значения класса не пересчитываются на лету внутри каждого виджета.

Нужен другой подход

Регион, бренд или менеджер не образуют естественную последовательность. Для динамических рейтингов и top-N по текущей мере сначала нужно отдельно определить, какой порядок считается бизнес-правилом.

Идея расчета

Мера берет номер текущего класса и пересчитывает продажи в окне всех клиентов, чей номер класса меньше или равен этому номеру.

Показать подпись

В строке отчета используется понятное имя класса: Silver, Gold, Platinum, Titanium.

Определить порядок

Сравнение идет по числу Customer Class Number, а не по алфавиту и не по текстовой подписи.

Собрать окно

FILTER ( ALL ( Customer ), ... ) возвращает все классы от первого до текущего включительно.

Посчитать продажи

В новом окне снова вычисляется сумма продаж, поэтому Gold включает продажи Silver и Gold.

Модель данных

Минимальная модель состоит из таблицы клиентов и таблицы продаж. Таблица с порогами классов полезна для объяснения правил классификации, но сама накопительная мера опирается на класс и его номер в Customer.

Элемент Зачем нужен Важно при подготовке
Customer[Customer Id] Ключ клиента и сторона связи с продажами. Уникален в справочнике клиентов; по нему строится связь с Sales.
Customer[Customer Class] Пользовательская подпись класса в таблице или диаграмме. Подпись не должна содержать технические префиксы вроде 01 Silver, если сортировка настроена отдельной колонкой.
Customer[Customer Class Number] Числовой порядок класса для условия "меньше или равно". Silver = 1, Gold = 2, Platinum = 3, Titanium = 4. Пропуски допустимы, если порядок остается однозначным.
Sales[Customer Id] Ключ клиента в факте продаж. Связь идет от Sales к Customer; строки продаж без клиента не попадут в классификацию.
Sales[Amount] Аддитивная сумма, на базе которой строится Sales Amount. Отрицательные корректировки остаются в факте и включаются в накопление до расчета итогов.
Class Config[Min Sales], Class Config[Max Sales] Пороговая таблица, которая объясняет, какой диапазон продаж соответствует каждому классу. Используйте ее как правило подготовки или справочную расшифровку. В отчете важно показать пользователю границы классов.
Customer -> Sales label + sort key стабильная классификация сортировка без префиксов

Меры

Базовая мера показывает продажи в текущем контексте. Накопительная мера расширяет фильтр клиентов до всех классов, которые идут до текущего класса включительно.

Продажи, количество клиентов и накопительный итог

Sales Amount считает сумму факта. # Customers считает клиентов из справочника, поэтому клиент без продаж остается в своем классе. Sales Amount RT Class накапливает продажи по номеру класса.

Sales Amount :=
SUM ( Sales[Amount] )

# Customers :=
DISTINCTCOUNT ( Customer[Customer Id] )

Sales Amount RT Class :=
CALCULATE (
    SUM ( Sales[Amount] ),
    FILTER (
        ALL ( Customer ),
        Customer[Customer Class Number] <= MAX ( Customer[Customer Class Number] )
    )
)
Почему в фильтре участвует вся таблица Customer

Текущая строка таблицы фильтрует один класс клиента. Чтобы получить накопление, расчету нужно временно снять этот фильтр, вернуть все классы с меньшим или равным номером и уже в этом окне пересчитать продажи.

Если в той же таблице Customer есть дополнительные атрибуты, например регион или сегмент клиента, ALL ( Customer ) снимает и их. Когда такие фильтры должны сохраняться, лучше вынести класс клиента в отдельное измерение и снимать фильтр только с него.

Как читать результат

В каждой строке накопительный итог включает текущий класс и все предыдущие классы по Customer Class Number. Итоговая строка совпадает с общей суммой продаж.

Customer Class Sales Amount # Customers Sales Amount RT Class Как читать строку
Silver 4 600 3 4 600 Только продажи Silver.
Gold 31 600 3 36 200 Silver плюс Gold.
Platinum 132 000 2 168 200 Silver, Gold и Platinum.
Titanium 465 000 2 633 200 Все классы до последнего.
Total 633 200 10 633 200 В общем контексте последний номер класса равен Titanium.
!

Порядок строк должен совпадать с порядком расчета. Если визуализация сортирует подписи по алфавиту, накопительная линия станет трудной для чтения: значение будет рассчитано правильно, но соседние точки окажутся не в бизнес-последовательности.

Особые случаи

Эти ситуации лучше продумать до публикации дашборда, потому что они меняют ожидания пользователя от накопительного значения.

Видны не все классы

Если в виджете оставить только Gold и Titanium, значение Gold все равно может включать Silver, потому что формула восстанавливает весь справочник Customer.

Клиент без продаж

# Customers считает клиентов из Customer, а не только клиентов с фактами. Такой клиент увеличит количество в классе, но не добавит сумму продаж.

Корректировки продаж

Отрицательные строки в Sales[Amount] сначала попадают в сумму своего класса, а затем участвуют в накопительном итоге следующих классов.

Визуальные разрезы

Накопительный итог по классу хорошо читается там, где ось или строки сохраняют тот же порядок, который использует мера.

Хорошо работает

  • Customer Class в строках таблицы, отсортированный по Customer Class Number.
  • Таблица с тремя мерами рядом: Sales Amount, # Customers, Sales Amount RT Class.
  • Столбчатая диаграмма для продаж класса и линия для накопительного итога.
  • Отдельная справочная таблица с порогами Silver, Gold, Platinum и Titanium.

Может ввести в заблуждение

  • Алфавитная сортировка классов: пользователь увидит правильные числа в неправильной последовательности.
  • Фильтры по атрибутам из Customer, если ожидается, что они сохранятся в накопительной мере.
  • Разрез по региону или бренду без бизнес-порядка "до текущего значения включительно".
  • Динамический рейтинг клиентов, если номер класса меняется вместе с фильтрами виджета.

Практические условия реализации

Короткая формула работает надежно только тогда, когда порядок классов, связи модели и ожидания от фильтров описаны явно.

Сортировочный ключ обязателен. Условие "меньше или равно" должно сравнивать числа, а не названия классов.

Классификацию лучше фиксировать заранее. Если клиент меняет класс при каждом выборе периода, результат становится зависимым от правил пересчета рейтинга.

ALL ( Customer ) снимает фильтры клиента. Для моделей с несколькими атрибутами клиента часто чище использовать отдельное измерение класса.

Итоговая строка не является суммой строк RT Class. Она пересчитывает меру в общем контексте и показывает накопление до последнего класса.
