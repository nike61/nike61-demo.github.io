Cumulative total

Basic scenario

Базовый накопительный итог показывает сумму всех операций от начала доступной истории до последней даты в текущей строке отчета. Он нужен, когда важно видеть не только оборот периода, но и накопленное состояние на дату.

сквозное накопление Sales Amount RT RT Weekdays будущие даты пустые

Как читается итог

Серые столбцы показывают продажи отдельных месяцев. Синяя линия складывает все месяцы от начала истории до текущей точки и не начинает счет заново в январе.

Sales Amount Sales Amount RT

Какую задачу решает

Обычная сумма в строке месяца отвечает на вопрос "сколько было в этом месяце". Накопительный итог отвечает на другой вопрос: "сколько уже набрано к концу этого месяца или дня".

Рост результата

Пользователь видит, как общий объем продаж, списаний, поступлений или других операций растет от даты к дате.

Состояние из движений

Для складских и финансовых сценариев состояние на дату можно получить из исходных транзакций, не создавая снимок на каждый день.

Контроль отсутствия фактов

День без продаж может сохранить предыдущий накопленный итог, а будущий период должен остаться пустым, чтобы не создавать ложный хвост графика.

Когда применять

Паттерн подходит для аддитивных фактов, которые можно складывать по времени, и для отчетов, где накопление идет от начала истории до текущей видимой даты.

Хороший сценарий

Продажи, выручка, количество операций, складские поступления и списания. Модель содержит полноценную таблицу Date, а факт связан с ней по дате операции.

Нужен другой подход

Если итог должен сбрасываться на границе года, квартала или месяца, используйте period-to-date расчеты. Если показатель является снимком состояния, например остатком на конец дня, нужны полуаддитивные паттерны.

Идея расчета

Мера заменяет текущий фильтр календаря на окно всех дат от начала таблицы Date до последней даты, видимой в строке отчета.

Найти конечную дату

Текущий контекст задает последнюю видимую дату через MAX ( 'Date'[Day Index] ).

Проверить факты

Если первая дата строки позже последней даты с продажами, мера возвращает blank.

Построить окно

FILTER ( ALL ( 'Date' ), ... ) оставляет все даты с индексом не больше конечного.

Посчитать сумму

В новом окне вычисляется та же базовая мера Sales Amount.

Модель данных

Минимальная модель состоит из календаря и таблицы транзакций. Календарь должен быть непрерывным: одна строка на каждый день без пропусков.

Элемент Зачем нужен Важно при подготовке
Date[Date] Календарная дата и ключ связи с фактом. Уникальная дата в каждой строке, диапазон покрывает всю историю анализа.
Date[Day Index] Непрерывный числовой индекс дня для условий "меньше или равно". Индекс растет на 1 каждый день и сортирует даты в естественном порядке.
Date[Year Month], Date[Date Label] Подписи для месячной и дневной детализации в дашборде. Текстовые подписи сортируются техническими ключами, без числовых префиксов в названии.
Date[Day of Week], Date[Day of Week Sort] Нужны, если пользователь анализирует будние дни отдельно от выходных. Для расчета только по будним дням фильтр дня недели должен быть явно возвращен в окно накопления.
Sales[Order Date] Дата операции и сторона связи Sales -> Date. Связь строится с Date[Date]; тип и гранулярность дат должны совпадать.
Sales[Order Day Index] Помогает найти последнюю дату, на которую есть продажи. Значение соответствует Date[Day Index] для даты заказа.
Sales[Amount] Аддитивная сумма, на базе которой строятся меры. Колонка факта называется иначе, чем мера Sales Amount, чтобы не было конфликта имен.
непрерывный календарь одна дата операции Sales -> Date сортировка подписей

Меры

В Visiology формулы записаны без VAR. Вместо сравнения DateTime используется Day Index, а пустое значение для будущих периодов возвращается через DIVIDE ( 0, 0 ).

Базовая сумма и сквозной накопительный итог

Sales Amount показывает продажи в текущем контексте. Sales Amount RT расширяет календарный фильтр до всех дат не позже последней видимой даты и поэтому накапливает сумму с начала истории.

Sales Amount :=
SUM ( Sales[Amount] )

Sales Amount RT :=
IF (
    MIN ( 'Date'[Day Index] )
        <= CALCULATE (
            MAX ( Sales[Order Day Index] ),
            REMOVEFILTERS ( 'Date' ),
            REMOVEFILTERS ( Sales )
        ),
    CALCULATE (
        SUM ( Sales[Amount] ),
        FILTER (
            ALL ( 'Date' ),
            'Date'[Day Index] <= MAX ( 'Date'[Day Index] )
        )
    ),
    DIVIDE ( 0, 0 )
)
Накопление только по будним дням (RT Weekdays)

Обычная мера Sales Amount RT снимает фильтры с таблицы Date, поэтому выбор дня недели на той же таблице не ограничивает накопительное окно. RT Weekdays явно возвращает условие Monday-Friday через сортировочный номер дня недели.

RT Weekdays :=
IF (
    MIN ( 'Date'[Day Index] )
        <= CALCULATE (
            MAX ( Sales[Order Day Index] ),
            REMOVEFILTERS ( 'Date' ),
            REMOVEFILTERS ( Sales )
        ),
    CALCULATE (
        SUM ( Sales[Amount] ),
        FILTER (
            ALL ( 'Date' ),
            AND (
                'Date'[Day Index] <= MAX ( 'Date'[Day Index] ),
                'Date'[Day of Week Sort] IN { 1, 2, 3, 4, 5 }
            )
        )
    ),
    DIVIDE ( 0, 0 )
)

Для произвольного выбора дней недели лучше не вшивать список значений в меру. Используйте отдельное измерение для такого атрибута или подготовьте бизнес-признак, например Is Working Day.

Как читать результат

Накопительная мера зависит от последней видимой даты. На уровне месяца она показывает итог на конец месяца, на уровне дня - итог на конец дня.

Период Что показывает Sales Amount Что показывает Sales Amount RT
Январь Продажи января. Продажи января.
Февраль Только продажи февраля. Январь плюс февраль.
Март Только продажи марта. Январь, февраль и март.
Январь следующего года Продажи этого января. Все продажи с начала истории, включая прошлый год.
Будущий месяц после последней продажи Пусто. Пусто, а не повтор последнего накопленного значения.
!

Годовая граница не сбрасывает расчет. Это сквозной накопительный итог с начала истории. Если в январе нужен новый старт, используйте расчет с началом периода, например YTD.

Особые случаи

Эти ситуации стоит проверить перед публикацией дашборда, потому что они чаще всего вызывают неверное чтение накопительной суммы.

День без продаж

Sales Amount будет пустым, но Sales Amount RT может сохранить накопленный итог предыдущих дней.

Частично заполненный месяц

Последний месяц остается видимым, если в нем уже есть продажи. Даты после последней продажи должны возвращать blank.

Фильтр дня недели

Sales Amount реагирует на выбор будних дней. Sales Amount RT его не сохраняет, а RT Weekdays сохраняет явно.

Визуальные разрезы

Накопительный итог лучше всего читается на оси, где сохраняется естественный порядок дат и понятно, какая точка является конечной.

Хорошо работает

  • Year Month для помесячного роста накопленного результата.
  • Date Label для дневного анализа и проверки дней без продаж.
  • Продукт, регион, канал продаж или клиентский сегмент как бизнес-фильтры поверх календаря.
  • Дни недели, если используется мера, которая явно сохраняет выбранные дни.

Может ввести в заблуждение

  • Разрез только по дню недели для Sales Amount RT: мера снимает фильтры с Date и считает календарное окно.
  • Годовая таблица, если пользователь ожидает сброс на каждом новом году.
  • Карточка без подписи периода: она показывает итог на последнюю дату текущего контекста, а не самостоятельный тип показателя.
  • Будущие месяцы на графике, если blank заменяется нулем или последним известным значением.

Практические условия реализации

Формула короткая, но ее смысл полностью зависит от календаря, связи с фактом и того, какие фильтры даты должны сохраняться.

Календарь задает начало истории. Если таблица Date начинается раньше нужного аналитического периода, итог будет накапливаться с более ранней даты.

Фильтр окна снимает фильтры с Date. Атрибуты той же таблицы, например день недели, нужно возвращать в расчет явно или выносить в отдельные измерения.

ALL используется внутри FILTER. Такая форма строит таблицу дат для окна накопления и хорошо читается в пользовательской мере.

Пустое значение лучше нуля для будущих периодов. Blank не выглядит как падение продаж и не продолжает линию там, где фактов еще нет.
