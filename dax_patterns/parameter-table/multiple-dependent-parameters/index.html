Multiple dependent parameters

Паттерн помогает задать несколько связанных настроек расчета так, чтобы пользователь видел только допустимые комбинации. Например, чем выше минимальное количество товаров в заказе, тем большую скидку можно выбрать.

Сценарий: порог 3, скидка 10%

При выбранном пороге 3 список скидок заканчивается на 30%, потому что в таблице параметров нет строк с более высокой скидкой для этого порога.

Какую задачу решает

В сценарном анализе часто есть несколько настроек, но не каждая пара значений имеет бизнес-смысл. Коммерческая команда может разрешать большую скидку только для более крупного заказа. Паттерн хранит допустимые пары в одной таблице параметров и не дает пользователю собрать сценарий, которого нет в правилах.

Срезы подсказывают допустимый выбор

Когда пользователь выбирает Min Quantity = 3, срез скидки показывает только значения до 30.00%. Запрещенные пары не нужно обрабатывать отдельной ошибкой в мере.

Одна таблица вместо многих правил

Discount содержит по одной строке на допустимую комбинацию порога и скидки. Эта структура проще, чем набор отдельных мер под каждый сценарий.

Продажи не отфильтровываются

Таблица параметров остается без связей с Sales. Она передает выбранные числа в меру, но не меняет состав строк продаж.

Когда применять

Паттерн подходит, когда расчет зависит от нескольких параметров с ограниченным списком значений, а допустимость одного значения зависит от другого. Это может быть порог количества и скидка, уровень риска и лимит, тариф и доступный коэффициент.

Хороший сценарий

Список допустимых комбинаций небольшой, заранее известен и его можно хранить как обычную таблицу набора данных. Пользователь должен выбирать сценарий, а не фильтровать факты.

Нужен другой подход

Если все комбинации допустимы, проще использовать независимые таблицы параметров. Если выбор должен реально отбирать продажи, регионы или продукты, используйте обычное измерение со связью.

Идея расчета

Связь между параметрами находится в строках самой таблицы Discount. Два среза используют разные колонки одной и той же таблицы, поэтому выбор порога автоматически оставляет только строки с разрешенными скидками.

Собрать комбинации

В Discount добавляются только допустимые пары Min Quantity и Discount.

Оставить без связей

Таблица не соединяется с Sales, Product или другими бизнес-таблицами.

Прочитать выбор

Служебные меры возвращают выбранный порог и выбранную скидку, а при неоднозначном выборе используют значения по умолчанию.

Посчитать по заказам

Discounted Amount группирует продажи по заказам и применяет скидку, если количество достигло порога.

Модель данных

Минимальная модель содержит факт продаж, измерение продукта и одну таблицу зависимых параметров. Номер заказа обязателен: именно на уровне заказа проверяется условие применения скидки.

Элемент Зачем нужен Важно при подготовке
Sales[Order Number] Гранулярность применения скидки. Один заказ может состоять из нескольких строк продаж и нескольких брендов. Мера сначала собирает строки текущего контекста по заказу, а затем проверяет суммарное количество.
Sales[Quantity] Количество товаров в строках заказа. Для сравнения с Min Quantity используется сумма количества по заказу.
Sales[Amount] Исходная сумма продаж до сценарной скидки. Sales Amount не зависит от параметров; параметры меняют только Discounted Amount.
Product[Brand] Бизнес-разрез для чтения результата по брендам. Обычная связь строится как Sales[ProductKey] -> Product[ProductKey].
Discount[Min Quantity] Порог количества товаров в заказе: например, от 1 до 10. Один порог может повторяться в нескольких строках, потому что ему соответствуют разные скидки.
Discount[Discount Label], Discount[Discount] Подпись скидки для интерфейса и числовая ставка для расчета: 0, 0,10, 0,30. Подпись удобно сортировать по числовой ставке, чтобы порядок был 0%, 5%, 10% и дальше по возрастанию.
Sales -> Product Discount без связей одна строка на допустимую пару Order Number обязателен Discount Label сортируется по Discount

Таблица зависимых параметров

Таблица Discount создается как физическая таблица модели. В примере она содержит пороги от 1 до 10 и скидки от 0% до 95% с шагом 5%, но оставляет только строки, где ставка не выше Min Quantity / 10.

Min Quantity Доступные значения Discount Label Как читать правило
1 0.00%, 5.00%, 10.00% Для минимального порога доступны только скидки до 10%.
3 0.00%, 5.00%, 10.00%, 15.00%, 20.00%, 25.00%, 30.00% Пара 3 и 50.00% отсутствует, поэтому такая скидка не появится в срезе.
10 0.00% ... 95.00% Высокий порог открывает весь список ставок из примера.
!

Discount остается таблицей параметров. Если связать ее с фактами или справочниками, выбор скидки начнет фильтровать данные, хотя должен только задавать числа для сценарной меры.

Меры

Служебные меры читают выбранные значения из зависимой таблицы. Для Min Quantity нельзя проверять единственную строку таблицы: один выбранный порог соответствует нескольким строкам с разными скидками. Поэтому используется проверка MIN = MAX по конкретной колонке.

Выбранный порог количества

Selected Min Quantity возвращает порог, если в текущем контексте осталось одно значение Discount[Min Quantity]. Если порог не выбран или выбран неоднозначно, применяется значение по умолчанию 1.

Selected Min Quantity :=
IF (
    MIN ( Discount[Min Quantity] ) = MAX ( Discount[Min Quantity] ),
    MAX ( Discount[Min Quantity] ),
    1
)
Выбранная скидка

Selected Discount возвращает числовую ставку при одном выбранном значении Discount[Discount]. Если скидка не выбрана или выбрано несколько ставок, расчет использует 0.

Selected Discount :=
IF (
    MIN ( Discount[Discount] ) = MAX ( Discount[Discount] ),
    MAX ( Discount[Discount] ),
    0
)
Базовые меры продаж

# Quantity и Sales Amount описывают исходные данные. Они нужны отдельно, чтобы показать: параметры не фильтруют факт продаж.

# Quantity :=
SUM ( Sales[Quantity] )

Sales Amount :=
SUM ( Sales[Amount] )
Сценарная мера Discounted Amount

Формула группирует строки текущего контекста по Sales[Order Number]. Для каждого заказа она считает количество и сумму через CALCULATE, а затем применяет выбранную скидку только к заказам, достигшим порога.

Discounted Amount :=
SUMX (
    SUMMARIZE ( Sales, Sales[Order Number] ),
    IF (
        CALCULATE ( SUM ( Sales[Quantity] ) ) >= Discount[Selected Min Quantity],
        ( 1 - Discount[Selected Discount] ) * CALCULATE ( SUM ( Sales[Amount] ) ),
        CALCULATE ( SUM ( Sales[Amount] ) )
    )
)

Особые случаи

Эти правила лучше явно зафиксировать рядом с дашбордом, чтобы пользователь понимал, где заканчивается выбор сценария и начинается обычный разрез данных.

Граница включительная

Условие использует >= Min Quantity. Заказ с количеством 3 получает скидку при выбранном пороге 3. Если нужно правило "строго больше", условие меняется на >.

Неоднозначный выбор

Если по одному параметру осталось несколько значений, только этот параметр возвращается к значению по умолчанию: порог к 1, скидка к 0. Второй параметр продолжает работать по своему выбору.

Итоговый контекст

Если один заказ содержит товары разных брендов, в строках брендов он может не пройти порог, но пройти его в общем итоге. Это нормальный эффект расчета на уровне заказа.

i

Проверяйте выбор по колонке, а не по строке таблицы. При Min Quantity = 3 в таблице остается несколько строк с разными скидками, поэтому проверка единственной строки ошибочно посчитает выбор неоднозначным.

Визуальные разрезы

Лучше всего паттерн читается как два одиночных выбора рядом с таблицей, KPI или графиком. Сначала пользователь выбирает порог, затем видит доступные ставки скидки для этого порога.

Хорошо читается

  • одиночный выбор Discount[Min Quantity] и зависимый выбор Discount[Discount Label];
  • таблица Brand x Sales Amount x Discounted Amount;
  • таблица допустимых пар Min Quantity x Discount Label для объяснения правил;
  • детализация по заказам для проверки, какие заказы прошли порог.

Может ввести в заблуждение

  • показывать множественный выбор параметра без объяснения значения по умолчанию;
  • считать итоговую строку простой суммой строк брендов, когда один заказ распределен между брендами;
  • использовать только Discount[Discount Label] как бизнес-разрез продаж без выбранного порога;
  • добавлять связь от Discount к фактам или справочникам.
