Multiple independent parameters

Паттерн помогает моделировать сценарии, где пользователь выбирает несколько независимых настроек: например, минимальный размер заказа для скидки и сам процент скидки. Выбор меняет только расчет меры, а не отбирает строки продаж.

Какую задачу решает

Бизнес-пользователь часто хочет проверить несколько управленческих сценариев на исторических данных: с какого размера заказа начинать скидку, какой процент дать, как изменится выручка по брендам. Создавать отдельную меру под каждую комбинацию неудобно, а связанные справочники начали бы фильтровать продажи вместо симуляции.

Сценарии без размножения мер

Discounted Amount остается одной мерой. Пользователь меняет только значения параметров, а формула сама пересчитывает результат.

Параметры независимы

Min Quantity не ограничивает список скидок, а Discount не ограничивает список порогов. Любая комбинация остается доступной.

Продажи не отфильтровываются

Таблицы параметров не связаны с Sales. Они дают скалярные значения для DAX-меры, но не меняют набор строк факта.

Когда применять

Паттерн подходит для расчетов, где несколько пользовательских настроек влияют на одну формулу независимо друг от друга. Это может быть порог и ставка скидки, горизонт прогноза и коэффициент роста, базовый сценарий и уровень риска.

Хороший сценарий

Каждый параметр можно выбрать сам по себе, а мера должна принять все выбранные значения и пересчитать показатель. Пользователь ожидает симуляцию, а не фильтрацию фактов.

Нужен другой подход

Если один выбор должен ограничивать доступные значения другого, используйте модель зависимых параметров. Например, если ставка скидки допустима только для конкретного порога количества, пары значений должны храниться в связанной логике комбинаций.

Идея расчета

Для каждого независимого выбора создается отдельная таблица параметров. Меры читают выбранные значения из этих таблиц и применяют их внутри расчета, а сама модель продаж остается обычной моделью с фактами и измерениями.

Создать параметры

Min Quantity хранит пороги количества, Discount хранит пользовательскую подпись и числовую ставку.

Оставить без связей

Таблицы параметров не соединяются с Sales, Product или другими бизнес-таблицами.

Прочитать выбор

Служебные меры возвращают выбранный порог и скидку. Если выбор неоднозначен, применяется значение по умолчанию.

Посчитать по заказам

Discounted Amount группирует продажи по заказам и применяет скидку к каждому заказу отдельно.

Модель данных

Минимальная модель содержит факт продаж, измерение продукта и две самостоятельные таблицы параметров. Для сценария скидки особенно важен номер заказа: именно на этом уровне проверяется порог количества.

Элемент Зачем нужен Важно при подготовке
Sales[Order Number] Гранулярность применения скидки. Один заказ может содержать несколько строк продаж и несколько брендов. Мера сначала собирает строки текущего контекста по заказу, а уже потом проверяет порог количества.
Sales[Quantity] Количество товаров в строках заказа. Для сравнения с Min Quantity используется сумма количества по заказу в текущем контексте.
Sales[Amount] Исходная сумма продаж до сценарной скидки. Sales Amount не меняется от выбранных параметров; параметры влияют только на Discounted Amount.
Product[Brand] Бизнес-разрез для чтения результата по брендам. Обычная связь строится как Sales[ProductKey] -> Product[ProductKey].
Min Quantity[Min Quantity] Доступные пороги количества: например, 1, 6 и 10. Таблица остается без связей. Показывайте колонку пользователю как одиночный выбор сценария.
Discount[Discount Label], Discount[Discount] Подпись скидки для интерфейса и числовая ставка для расчета: 0, 0,15 или 0,25. Подпись удобно сортировать по числовой ставке, чтобы порядок был 0%, 15%, 25%.
Sales -> Product Min Quantity без связей Discount без связей Order Number обязателен Discount Label сортируется по Discount

Таблицы параметров

В Visiology параметры удобно подготовить как обычные таблицы набора данных. Их небольшие списки значений обновляются вместе с моделью и остаются прозрачными для пользователя дашборда.

Min Quantity Как читать выбор
1 Скидка применяется почти ко всем заказам с положительным количеством.
6 Скидка применяется к заказам, где суммарно не меньше 6 товаров.
10 Скидка применяется только к более крупным заказам.
Discount Label Discount Как читать выбор
0.00% 0 Сценарий без скидки.
15.00% 0,15 Скидка 15% для заказов, прошедших порог.
25.00% 0,25 Скидка 25% для заказов, прошедших порог.
!

Параметры не должны становиться измерениями продаж. Если связать Min Quantity или Discount с фактами, выбор начнет отбирать строки данных, хотя должен только передавать значение в формулу.

Меры

Служебные меры превращают выбранные строки параметров в числа. Основная мера затем использует эти числа в расчете скидки по каждому заказу.

Выбранный порог количества

Selected Min Quantity возвращает порог, если в контексте ровно одна строка Min Quantity. Если выбора нет или выбрано несколько порогов, применяется значение по умолчанию 1.

Selected Min Quantity :=
IF (
    COUNTX ( 'Min Quantity', 'Min Quantity'[Min Quantity] ) = 1,
    MAX ( 'Min Quantity'[Min Quantity] ),
    1
)
Выбранная скидка

Selected Discount возвращает числовую ставку скидки при однозначном выборе. Если скидка не выбрана или выбрано несколько ставок, расчет использует 0.

Selected Discount :=
IF (
    COUNTX ( Discount, Discount[Discount] ) = 1,
    MAX ( Discount[Discount] ),
    0
)
Базовые меры продаж

# Quantity и Sales Amount описывают исходные данные. Они нужны отдельно, чтобы было видно: параметры меняют сценарную меру, а не сами продажи.

# Quantity :=
SUM ( Sales[Quantity] )

Sales Amount :=
SUM ( Sales[Amount] )
Сценарная мера Discounted Amount

Формула группирует строки текущего контекста по Sales[Order Number]. Для каждого заказа она проверяет суммарное количество и применяет выбранную скидку только если количество не меньше выбранного порога.

Discounted Amount :=
SUMX (
    SUMMARIZE ( Sales, Sales[Order Number] ),
    IF (
        CALCULATE ( SUM ( Sales[Quantity] ) ) >= 'Min Quantity'[Selected Min Quantity],
        ( 1 - Discount[Selected Discount] ) * CALCULATE ( SUM ( Sales[Amount] ) ),
        CALCULATE ( SUM ( Sales[Amount] ) )
    )
)

Особые случаи

Эти правила лучше явно зафиксировать в описании дашборда, чтобы пользователь правильно сравнивал сценарии и не путал параметры с фильтрами данных.

Граница включительная

Условие использует >= Min Quantity. Заказ с количеством 6 получает скидку при выбранном пороге 6. Если нужно правило "строго больше", условие меняется на >.

Неоднозначный выбор

Множественный выбор одного параметра возвращает только этот параметр к значению по умолчанию: порог к 1, скидку к 0. Второй параметр продолжает работать по своему выбору.

Итоговый контекст

Если один заказ содержит товары разных брендов, в строках брендов он может не пройти порог, но пройти его в общем итоге. Это нормальный эффект расчета на уровне заказа.

i

Параметры независимы только пока между ними нет ограничений. Если бизнес должен запретить некоторые пары "порог + скидка", храните допустимые комбинации явно и используйте паттерн зависимых параметров.

Визуальные разрезы

Лучше всего паттерн читается как пара одиночных переключателей рядом с таблицей, KPI или графиком. Разрезы по бренду, дате и категории остаются обычными бизнес-измерениями.

Хорошо читается

  • два одиночных выбора: Min Quantity и Discount Label;
  • таблица Brand x Sales Amount x Discounted Amount;
  • сетка сценариев Min Quantity x Discount Label, если нужно сравнить все комбинации;
  • детализация по заказам для проверки, какие заказы прошли порог.

Может ввести в заблуждение

  • показывать множественный выбор параметра без объяснения значения по умолчанию;
  • считать итоговую строку простой суммой строк брендов, когда один заказ распределен между брендами;
  • использовать Discount[Discount] как бизнес-измерение продаж;
  • добавлять связи от таблиц параметров к фактам или справочникам.
