Стандартные расчеты по времени

Filtering other date attributes

Паттерн нужен, когда накопительный расчет по датам нужно читать вместе с дополнительным свойством даты: днем недели, рабочим днем, праздником, сезоном или другим признаком.

YTD по дню недели атрибуты даты отдельная таблица атрибута будущие периоды скрываются

Где теряется смысл

Если разрез по дню недели берется из таблицы Date, календарное окно YTD снимает и этот фильтр. Отдельная таблица дня недели сохраняет фильтр на фактах.

фильтр снят вместе с Date фильтр сохранен отдельно

Когда применять

Паттерн помогает там, где пользователь ожидает увидеть накопительный итог не по всем датам, а по датам с конкретным свойством.

Хороший сценарий

Отчет использует стандартный календарь, аддитивную меру и разрезы вроде Day of Week, Working Day, праздника или сезона. Атрибут должен зависеть от той даты факта, по которой считается мера, например от Sales[Order Date].

Лучше выбрать другой подход

Нестандартные недели, календарь 4-4-5, плавающие периоды и сложные рабочие календари лучше сразу моделировать как отдельный календарь. Если пользователю нужен множественный выбор по атрибуту даты, надежнее вынести атрибут в отдельную таблицу, а не пытаться восстановить фильтр внутри одной меры.

Идея расчета

Обычная YTD-мера строит собственное календарное окно и для этого снимает фильтры с таблицы Date. Если дополнительный атрибут тоже лежит в Date, он снимается вместе с календарным фильтром.

Пользователь выбирает разрез

Например, строки таблицы показывают Date[Day of Week] рядом с кварталом или годом.

Мера открывает YTD-окно

FILTER ( ALL ( 'Date' ), ... ) заменяет текущие даты диапазоном с начала года.

Фильтр атрибута исчезает

Фильтр дня недели тоже находится в Date, поэтому строка Mon начинает считать все дни до последнего понедельника.

Фильтр нужно отделить

Либо восстановить одно значение атрибута в мере, либо вынести атрибут в отдельную таблицу, связанную с фактом.

Модель данных

Базовая часть модели такая же, как у остальных стандартных расчетов по времени: явная таблица дат, таблица фактов и связь по дате заказа. Для устойчивого разреза добавляется отдельная таблица атрибута даты.

Таблица и колонка Зачем нужна Как формируется
Date[Date] Календарная дата и ключ связи с Sales[Order Date]. Одна строка на каждый день нужного диапазона без пропусков.
Date[Year], Date[Year Quarter], Date[Year Month] Календарные уровни для строк, колонок и фильтров дашборда. Строятся из даты; текстовые подписи сортируются техническими ключами.
Date[Day Index] Непрерывная шкала дней для сравнения границ периода. Последовательный номер дня в календарной таблице.
Date[Year Start Day Index] Первый день календарного года для YTD-окна. Для каждой даты содержит Day Index 1 января того же года.
Date[Day of Week], Date[Day of Week Sort] Демонстрируют проблему: это удобный календарный атрибут, но его фильтр снимается YTD-мерой вместе с Date. День недели и числовая сортировка подписи.
Date[DateWithSales] Помогает не показывать значения после последней даты с фактами. 1, если дата не позже последней даты продаж; иначе 0.
Sales[Order Date], Sales[Order Day Index], Sales[Amount] Дата факта, ее дневной индекс и числовое значение продаж. Приходят из фактов или подготавливаются при загрузке данных.
Day of Week[Date], Day of Week[Day of Week], Day of Week[Day of Week Sort] Отдельный источник разреза по дню недели, который фильтрует Sales напрямую и не зависит от снятия фильтров с Date. Копия дат из Date с нужными атрибутами и сортировкой подписи.
Date[Date] -> Sales[Order Date] Day of Week[Date] -> Sales[Order Date] обе связи 1:* разрез по дню недели брать из отдельной таблицы

Меры

В примерах физическая колонка факта называется Sales[Amount], а пользовательская мера - Sales Amount. Это отделяет бизнес-меру от технической колонки.

Базовая мера и контроль будущих дат

Sales Amount показывает сумму продаж. ShowValueForDates - техническая мера: она возвращает 1, если первый день выбранного периода не позже последней даты продаж, и пустое значение для будущих периодов.

Sales Amount :=
SUM ( Sales[Amount] )

ShowValueForDates :=
IF (
    MIN ( 'Date'[Day Index] )
        <= CALCULATE (
            MAX ( Sales[Order Day Index] ),
            REMOVEFILTERS ( 'Date' ),
            REMOVEFILTERS ( Sales )
        ),
    1,
    DIVIDE ( 0, 0 )
)
Sales YTD: календарный итог с начала года

Эта мера корректна для календарных разрезов из таблицы Date. Но из-за ALL ( 'Date' ) она снимает и фильтры дополнительных атрибутов, если эти атрибуты находятся в той же таблице.

Sales YTD :=
IF (
    MIN ( 'Date'[Day Index] )
        <= CALCULATE (
            MAX ( Sales[Order Day Index] ),
            REMOVEFILTERS ( 'Date' ),
            REMOVEFILTERS ( Sales )
        ),
    CALCULATE (
        SUM ( Sales[Amount] ),
        FILTER (
            ALL ( 'Date' ),
            AND (
                'Date'[Day Index] >= MAX ( 'Date'[Year Start Day Index] ),
                'Date'[Day Index] <= MAX ( 'Date'[Day Index] )
            )
        )
    ),
    DIVIDE ( 0, 0 )
)
Sales YTD (day of week): восстановление одного фильтра дня недели

Мера повторно добавляет фильтр дня недели, если в текущем контексте выбран ровно один день недели. Для строк таблицы по Date[Day of Week] это дает ожидаемый YTD только по выбранному дню недели.

Sales YTD (day of week) :=
IF (
    MIN ( 'Date'[Day Index] )
        <= CALCULATE (
            MAX ( Sales[Order Day Index] ),
            REMOVEFILTERS ( 'Date' ),
            REMOVEFILTERS ( Sales )
        ),
    IF (
        MIN ( 'Date'[Day of Week Sort] ) = MAX ( 'Date'[Day of Week Sort] ),
        CALCULATE (
            SUM ( Sales[Amount] ),
            FILTER (
                ALL ( 'Date' ),
                AND (
                    AND (
                        'Date'[Day Index] >= MAX ( 'Date'[Year Start Day Index] ),
                        'Date'[Day Index] <= MAX ( 'Date'[Day Index] )
                    ),
                    'Date'[Day of Week Sort] = MIN ( 'Date'[Day of Week Sort] )
                )
            )
        ),
        CALCULATE (
            SUM ( Sales[Amount] ),
            FILTER (
                ALL ( 'Date' ),
                AND (
                    'Date'[Day Index] >= MAX ( 'Date'[Year Start Day Index] ),
                    'Date'[Day Index] <= MAX ( 'Date'[Day Index] )
                )
            )
        )
    ),
    DIVIDE ( 0, 0 )
)

Если в фильтре выбрано несколько дней недели из Date, эта формула осознанно возвращается к базовому Sales YTD. Для такого сценария используйте отдельную таблицу атрибута.

Отдельная таблица атрибута даты

Смысл модельного решения можно описать как копию дат с нужным атрибутом. В реальной модели такую таблицу удобно подготовить в источнике данных или в ETL и связать с фактом по той же дате заказа.

Day of Week =
SELECTCOLUMNS (
    'Date',
    "Date", 'Date'[Date],
    "Day of Week", 'Date'[Day of Week]
)

После этого визуальный разрез берется из Day of Week[Day of Week]. Базовая мера Sales YTD продолжает снимать фильтры только с Date, а фильтр отдельной таблицы остается на Sales.

Как читать результат

Главная проверка - сумма строк по дням недели должна сходиться с итогом YTD за период только тогда, когда строки действительно фильтруют факты по дню недели.

Базовый YTD в Date-разрезе

Строка дня недели выглядит правдоподобно, но фактически считает все дни недели до последней видимой даты этой строки. Такой результат нельзя читать как вклад выбранного дня недели.

Восстановленная мера

Sales YTD (day of week) подходит для строки, где виден один день недели. На итоговой строке или без фильтра дня недели она совпадает с обычным Sales YTD.

Отдельная таблица

Разрез из Day of Week работает с базовой YTD-мерой и лучше масштабируется на несколько атрибутов: рабочий день, праздник, сезон или тип торгового дня.

!

Следите за датой факта. Если отчет считает продажи по Order Date, то отдельная таблица атрибута должна фильтровать Sales[Order Date]. Для доставки, оплаты или другой даты нужна отдельная таблица атрибута, связанная с соответствующей датой факта.

Визуальные разрезы

Разрез должен отвечать на вопрос, какие даты попали в накопительное окно и какие факты остались после фильтрации.

Хорошо работает

  • Date[Year], Date[Year Quarter] и Date[Year Month] как календарная ось для YTD.
  • Day of Week[Day of Week] из отдельной таблицы для разреза по дням недели.
  • Обычные бизнес-фильтры по товару, региону, каналу или клиенту поверх календарного окна.
  • Таблица, где итог по дням недели можно сверить с YTD за тот же квартал или год.

Может запутать

  • Date[Day of Week] вместе с обычным Sales YTD, если пользователь ожидает вклад только выбранного дня недели.
  • Множественный выбор по Date[Day of Week] для меры Sales YTD (day of week): формула восстанавливает только одно значение дня недели.
  • Смешивание атрибутов разных дат факта, например дня недели заказа и календаря доставки в одном расчете.
  • Будущие месяцы после последней продажи. Такие периоды должны оставаться пустыми, а не тянуть последнее накопленное значение вперед.

Практические условия реализации

Паттерн остается простым, если заранее договориться, где живет каждый атрибут даты и какие разрезы пользователь будет применять в дашборде.

Календарное окно снимает фильтры с Date. Это ожидаемое поведение для YTD: мера заменяет текущий набор дат собственным диапазоном с начала года.

Один день недели можно восстановить в мере. Такой вариант удобен для простого табличного отчета, но его нужно явно документировать для итоговых строк и множественного выбора.

Несколько атрибутов лучше вынести в модель. Отдельная таблица может хранить день недели, признак рабочего дня, праздник и сезон, если все они зависят от одной даты факта.

Blank важнее нуля. Для будущих периодов используйте пустой результат через DIVIDE ( 0, 0 ), чтобы визуализация не показывала фиктивные продажи.
