Уникальные значения из связанных транзакций

Паттерн считает разные бизнес-значения из справочника только по тем элементам, которые действительно участвовали в операциях. Он помогает отвечать на вопросы про товары, клиентов или другие атрибуты, когда технический ключ не равен бизнес-сущности.

несколько факт-таблиц Product Name вместо ProductKey пересечение и объединение нулевые периоды

Что именно считается

Каждый круг показывает названия товаров, найденные через одну таблицу операций. Меры считают левую часть, правую часть, пересечение, объединение и разницу множеств.

есть в продажах есть в поступлениях есть в обеих таблицах

Какую задачу решает

Обычный подсчет строк отвечает на вопрос "сколько было операций". Подсчет ключей отвечает на вопрос "сколько технических записей встретилось". Этот паттерн отвечает на бизнес-вопрос: "сколько разных названий товаров было затронуто".

Считать бизнес-значения

Если одно название товара представлено несколькими ProductKey, продажи по этим ключам должны дать один товар в результате.

Сравнивать источники операций

Можно увидеть, какие названия есть в продажах, какие есть в поступлениях, какие есть в обоих потоках и какие встречаются хотя бы в одном.

Читать периоды корректно

Годовой результат считается на множестве названий за год, поэтому он не обязан равняться сумме месяцев.

Когда применять

Паттерн нужен, когда аналитический объект живет в справочнике, а факт-таблицы только показывают, что этот объект участвовал в операциях.

Хороший сценарий

Есть одна или несколько факт-таблиц, связанные со справочником, и нужно считать уникальные значения атрибута справочника: название товара, бренд, группу клиента или другой бизнес-признак.

Лучше упростить

Если целевой столбец уже находится в факт-таблице на нужной гранулярности и технический ключ совпадает с бизнес-сущностью, обычно достаточно обычного DISTINCTCOUNT.

Идея расчета

Вместо прямых операций над множествами мера проходит по уникальному списку названий и проверяет, есть ли у каждого названия активность в продажах и поступлениях в текущем контексте отчета.

Выбрать атрибут

Определяем бизнес-значение, которое нужно считать: например Product[Product Name].

Собрать список

Создаем справочник Product Names с одной строкой на каждое уникальное название.

Отметить активность

В таблице Product Name Activity храним количество строк продаж и поступлений для пары дата + название.

Сложить признаки

SUMX проходит по названиям и добавляет 1, если условие меры выполнено.

Модель данных

Минимальная реализация использует две исходные факт-таблицы и одну вспомогательную таблицу активности. Она делает расчет устойчивым для месяцев без операций и для названий, которые есть только в одном источнике.

Элемент Зачем нужен Важно при подготовке
Date[Date], Date[Calendar Year], Date[Year Month] Задают календарный контекст для строк отчета и фильтров. Календарь должен покрывать анализируемые даты продаж и поступлений; подписи месяцев сортируются техническим ключом.
Product[ProductKey], Product[Product Name] Справочник товаров и связь исходных операций с товаром. Product Name может повторяться у разных ключей, поэтому считать нужно название, а не ключ.
Product Names[Product Name] Уникальный список названий, по которому итерируются меры. Таблица связана с Product и Product Name Activity по названию; пользовательские подписи не требуют числовых префиксов.
Sales и Receipts Исходные таблицы операций: продажи и поступления. Обе таблицы связаны с Date и Product. Они используются для подготовки активности, а не напрямую для логики множеств в мерах.
Product Name Activity[Sales Rows], [Receipt Rows] Показывают, были ли продажи или поступления для конкретного названия в конкретную дату. Таблица строится на гранулярности Date x Product Name; строки без операций получают нули, чтобы пустые месяцы давали 0.
Date x Product Name уникальный список названий флаги активности нулевые строки для пустых дат

Меры

Все меры считают названия товаров, а не строки операций. Разница между ними только в условии, которое проверяется для каждого названия в текущем периоде.

Названия, найденные в продажах

# Prods from Sales добавляет 1 для каждого названия, у которого сумма Sales Rows больше нуля.

# Prods from Sales :=
SUMX (
    'Product Names',
    IF (
        CALCULATE ( SUM ( 'Product Name Activity'[Sales Rows] ) ) + 0 > 0,
        1,
        0
    )
)
Названия, найденные в поступлениях

# Prods from Receipts использует ту же логику, но проверяет строки поступлений.

# Prods from Receipts :=
SUMX (
    'Product Names',
    IF (
        CALCULATE ( SUM ( 'Product Name Activity'[Receipt Rows] ) ) + 0 > 0,
        1,
        0
    )
)
Названия, найденные в обеих таблицах

# Prods from Both считает название только тогда, когда в текущем контексте есть и продажи, и поступления.

# Prods from Both :=
SUMX (
    'Product Names',
    IF (
        CALCULATE ( SUM ( 'Product Name Activity'[Sales Rows] ) ) + 0 > 0,
        IF (
            CALCULATE ( SUM ( 'Product Name Activity'[Receipt Rows] ) ) + 0 > 0,
            1,
            0
        ),
        0
    )
)
Названия, найденные хотя бы в одной таблице

# Prods from Any считает объединение: продажа или поступление достаточно, чтобы название попало в результат.

# Prods from Any :=
SUMX (
    'Product Names',
    IF (
        CALCULATE ( SUM ( 'Product Name Activity'[Sales Rows] ) )
            + CALCULATE ( SUM ( 'Product Name Activity'[Receipt Rows] ) ) + 0 > 0,
        1,
        0
    )
)
Названия из продаж без поступлений

# Prods in Sales and not in Receipts показывает разницу множеств: название продавалось, но не поступало в выбранном контексте.

# Prods in Sales and not in Receipts :=
SUMX (
    'Product Names',
    IF (
        CALCULATE ( SUM ( 'Product Name Activity'[Sales Rows] ) ) + 0 > 0,
        IF (
            CALCULATE ( SUM ( 'Product Name Activity'[Receipt Rows] ) ) + 0 = 0,
            1,
            0
        ),
        0
    )
)

Как читать результат

Значение в строке периода считается по множеству названий внутри этого периода. Если одно и то же название встречается несколько раз или связано с несколькими ключами товара, оно все равно дает единицу.

Year Month # Prods from Sales # Prods from Receipts # Prods from Both # Prods from Any # Prods in Sales and not in Receipts

2023-02 4 4 2 6 2
2023-03 3 0 0 3 3
2024-01 3 4 3 4 0
2024-02 0 2 0 2 0
2024-03 0 0 0 0 0
!

Год не складывается из месяцев. Например, годовой # Prods from Both за CY 2023 равен 5, хотя в январе и феврале по 2. Название может иметь продажу в одном месяце, поступление в другом и попасть в пересечение только на уровне года.

Разрезы в дашборде

Расчет корректен для тех разрезов, которые представлены в гранулярности вспомогательной таблицы или фильтруют ее через связи модели.

Имеют смысл

  • Date[Year Month] и Date[Calendar Year] для анализа по времени.
  • Product Names[Product Name], чтобы увидеть sales-only, receipt-only и названия без операций.
  • Атрибуты, добавленные в ту же гранулярность активности: например склад, регион или канал, если расчет действительно нужен в таком разрезе.

Требуют отдельной подготовки

  • Разрезы, которые есть только в Sales или только в Receipts, не описывают оба множества одновременно.
  • Если нужен анализ по клиенту, складу или магазину, добавьте этот ключ во вспомогательную таблицу и пересоберите признаки активности.
  • Не суммируйте месячные значения вручную для годового итога: повторяющиеся названия будут посчитаны несколько раз.

Важные случаи

Большинство ошибок возникает не в формуле, а в ожиданиях от гранулярности и фильтров.

Дубликаты названий

Два разных ProductKey с одним Product Name считаются как одно название. Это ожидаемое поведение паттерна.

Пустые периоды

Месяц без продаж и поступлений должен показывать нули. Для этого в активности нужен полный каркас дат и названий.

Только поступления

Название, которое есть только в Receipts, входит в # Prods from Any, но не входит в # Prods from Sales.
